# 既知の落とし穴台帳（追記型）

「実物が届くまで気づけない」種類のミスを蓄積する台帳。
pcb-preorder-checks の実行前に必ず読み、実行中に新種を見つけたら追記する。
1 エントリ = 1 教訓。発生源（自分の失敗 / 外部の公開事例）を書く。

書式:

```
## <slug>
- 症状: 実物で何が起きるか
- 原因: 設計データ上でなぜ検出されなかったか
- 発注前チェック: どの確認で防げるか
- 出典: 事例への参照
```

---

## ffc-contact-side

- 症状: FFC/FPC ケーブルを挿してもディスプレイ等が動作しない。コネクタと
  フレックスの接点面が逆で電気的に接触していない
- 原因: 回路図・ネットリスト・DRC はすべて正しい。接点面（上接点/下接点）と
  フリップ有無はネットリストに現れない物理属性のため、機械検証をすり抜ける
- 発注前チェック: コネクタのデータシートの接点面と、相手側フレックスの
  接点面・折り返しの有無を突き合わせる（preorder-checklist「極性・向き」節）。
  設計ブリーフ段階で接点面を明示するのが最善（pcb-design-brief テンプレート）
- 出典: @gclue_akira 実例
  <https://x.com/gclue_akira/status/2075365960018702437>（2026-07）

## schematic-fw-pinmap-divergence

- 症状: 基板は正常に製造・実装されたのに、ボタン等の入力が FW から認識されない
- 原因: 回路図のピン割当と FW のピンマップが別々に作られ、それぞれ単独では
  「正しく見える」。突き合わせをしないと不一致が実機まで潜伏する
- 発注前チェック: 回路図のネット名 ⇔ FW ピンマップの突き合わせ表を作る
  （preorder-checklist「FW 整合」節）。回路図から FW を生成する場合も、
  生成後に逆方向の照合をする
- 出典: @gclue_akira 実例（Opus4.8 設計基板で Display は動作、ボタン不認識）
  <https://x.com/gclue_akira/status/2074614329597628872> 前後のスレッド（2026-07）

## footprint-package-mismatch

- 症状: 実装しようとすると部品がパッドに載らない/はんだが付かない。
  ピッチ違い・パッケージ違いのフットプリントで製造されている
- 原因: ネットリスト・DRC・実装プレビューはフットプリントを「正」として
  動くため、フットプリント自体が発注部品と寸法不整合でもすべて正しく見える。
  ジェネリック名（"SOP-8" 等）のフットプリント流用と 0402/0603 取り違えが典型
- 発注前チェック: 主要部品のフットプリント寸法を C 番号のデータシートの
  パッケージ図と突き合わせる（preorder-checklist「フットプリント↔実部品の
  整合」節）
- 出典: PCBA 発注の最頻失敗類型（スモークレビューでの指摘により追加）

## order-form-mismatch

- 症状: 4 層設計が 2 層で製造される、板厚違いで嵌合部品が合わない、
  NPTH 指定の取付穴がめっき付きで届く
- 原因: JLCPCB の発注フォーム（層数・板厚・銅厚・表面処理）は設計データから
  自動では埋まらず、フォーム値の誤りは設計側のどの検証にも現れない
- 発注前チェック: フォームの各値を設計値と 1 対 1 で照合する
  （preorder-checklist「製造制約・コスト」節）
- 出典: 予防的エントリ（スモークレビューでの指摘により追加）

## dnp-not-marked

- 症状: 手持ち在庫を使う予定の部品が PCBA でも実装されて届く（二重部品・
  費用増）。または逆に、実装されるべき部品が DNP 扱いで欠品する
- 原因: 「自分の在庫を使う」という方針が BOM/CPL の DNP フラグに反映されて
  いない。アップロードは通り、プレビューでも「載っている」こと自体は
  正しく見える
- 発注前チェック: DNP 方針の部品リストを作り、実装プレビューに載って
  **いない**ことを確認する（cpl-placement-review「全部品の目視」節）
- 出典: pcb-design-brief の在庫方針欄から派生する典型リスク（予防的エントリ）
