# pitfalls.md — Fusion API を使う前に読む罠カタログ

出典の凡例: 「実測済み 2026-07-03」= `research/official-mcp-probe.md` のスモークテストで実際に検証済み。それ以外は出典名を付したコミュニティ報告（`research/*.md` を参照）。複数ソースで独立に報告されたもののみ採録している（`SYNTHESIS.md` §3 が一次資料）。

## 単位・座標系

- **Fusion Python API の内部長さ単位は常に cm**（表示設定と無関係）。mm のつもりで値をそのまま渡すと10倍の事故になる。50mm を意図するなら API には `5.0` を渡す。**実測済み 2026-07-03**: 10mm 立方体を作る際 `side_cm = 1.0` と明示変換し、バウンディングボックスが `dx=dy=dz=1.0cm = 10.0mm` になることを確認した。ツールの `inputSchema` には単位の注記が一切ないため、変換を忘れても実行時エラーにはならず、10倍の寸法違いとして静かに成立してしまう。
- **XZ/YZ 平面では Z 軸の符号が反転する**。平面選択後は必ず操作前後で座標を検証すること（rahayesj `SPATIAL_AWARENESS.md`、日本語記事でも同種の事故が複数報告）。
- 単位・寸法はプロンプト/スクリプトコメントで必ず明示する。「mm と指示しても cm で生成される」事故が公式 MCP でも報告されている（note.com/mar810）。

## スケッチ・フィーチャー操作

- スケッチ平面には B-Rep 面より**コンストラクション平面**の方が安定する（Zenn/ux_xu の限界検証連載）。
- 拘束は「中点拘束」より「**点-点間距離寸法**」の方が AI 生成コードに対して堅牢（同上）。
- プロファイル選択則は**文脈依存**（同上、E2Eテストで誤誘導リスクを確認）: 「面積最小のものを選ぶ」ルールが有効なのは**内側の穴のような切り抜き対象を選びたい場合**。外形（ボディ本体）を押し出したい場合は逆に**面積最大**、あるいは既知の頂点座標などによる明示的な位置判定が必要になる。1枚のスケッチに外形と穴を同居させて `profiles` が複数返る構成では、常に面積最小を採用すると誤って穴側のプロファイルを外形として押し出しかねない。用途（切り抜きか外形か）を先に決めてから採用するルールを選ぶこと。
- **revolve はプロファイルの一部でも回転軸（x≥0 と x≤0）をまたぐと失敗する**（ndoo `CLAUDE.md`）。ただし**プロファイルが回転軸に「接する」だけなら成功する**: 軸上に辺・端点を持つプロファイル（例: (0,0)–(1.0,0) を含む輪郭）の 360° revolve は問題なく通ることを確認した（**実測 2026-07-06**。ndoo 報告の「端点が原点上でも拒否」は再現せず。またぐ=NG / 接する=OK が境界）。円弧は `addByCenterStartSweep`（回転方向が不定になりがち）でなく `addByThreePoints` を使う方が安全（ndoo `CLAUDE.md`）。
- **`profiles.count == 0` になったときのチェックリスト**（ndoo `CLAUDE.md`）:
  1. 開いた輪郭（端点が未接続）
  2. 重複エッジ（面積ゼロの領域ができている）
  3. 回転軸上に両端点がある
  4. 間違ったスケッチ平面を参照している
- 穴あけは「3D 円柱を後から引き算」すると歪みやすい。**2D 断面で先に円を抜いてから押し出す**方が安定する（Qiita/issey_dotlog）。
- 薄壁付近の `fillet` は失敗しやすい。`chamfer` の方が安定する（Zenn/optimisuke）。

## モード・スレッド・状態管理

- **パラメトリック/ダイレクトモードを操作前に明示確認する**。`TemporaryBRepManager` で作ったボディはパラメトリックモードでは `BaseFeature` でラップしないとタイムラインに乗らない。undo の挙動もモード依存（faust-machines/ndoo の実装知見。faust-machines は `undo` ツールに専用の設計モード安全ガードを実装している）。
- **Fusion API はメインスレッド限定**。UI スレッド外からの API 呼び出しはクラッシュ要因になる。アドイン実装は `CustomEvent`/Work Queue で明示的にメインスレッドへディスパッチしている（faust-machines, AuraFriday）。公式 MCP の `fusion_mcp_execute` script 経由の実行では、この制約はサーバ側が担保している前提で使う。
- **undo の粒度管理が事故防止の要**。大きすぎる undo（意図した操作以上を巻き戻す）も、小さすぎる undo（部分的にしか戻らず不整合な状態が残る）もどちらも危険。1操作＝1確認のサイクルを崩さない。
- 重要ファイルには直接触らせない。**新規ドキュメントかコピーで作業する**（`SKILL.md` §2③、`setup.md` 冒頭のガードレールと同一の原則）。
- **`script` の例外処理方針**: ツール説明文自体が「`run` 関数内で例外をキャッチするな」と明記している（実測済み 2026-07-03、`tools.md` 参照）。デバッグ性重視の設計であり、呼び出し側も try/except で握りつぶさず、エラーをそのまま報告すること。

## 製造ドメイン

- **材料・製造プロセスを最初に宣言しないと、AI はデフォルトで金属 CNC 的な公差を切り、3D プリント後に組立不能になる**（knightli.com の惑星歯車インデクサー改造実例）。推奨プロンプト文例:
  > "This model is for FDM 3D printing, using PLA. The goal is to install a 6mm bearing, so printing tolerance and press fit should be considered. Do not handle it as CNC metal machining tolerance."
- 材料は規格レベルまで明示すると精度が上がる（「鋼」でなく「AS/NZS 1734 Al 5052-H32, 2mm 厚」のように）。
- **AI 編集結果は画面表示（視覚）だけでは判断できない**（knightli.com）。0.1mm オーダーの公差が機能（ベアリングの回転滑らかさ等）を左右するケースでは、実物プリント・組立による物理検証が必須。歯車が実際には回らなかった実例が複数ソースで報告されている。`SKILL.md` §4 の三段検証・物理検証フラグに直結する。
- 適性の見極め: プリズマティック形状・板金・パラメトリック形状は得意、有機曲面は不向き（「犬のバッジ画像を同じ形に」は完全失敗という報告あり）。

## Electronics / テキストコマンドの罠

出典: `projects/arduino-uno-compat/comparison/fusion-electronics-run.md`（実測済み 2026-07-03、Fusion Electronics の書き込み可否を専用調査）。

- **引数不足・名前解決不能な `Electron.<cmd>`（`executeTextCommand("Electron.run <eagle cmd>")` 経由）はモーダルダイアログを開き、MCP サーバ全体を約2分ハングさせる**。実測: `Electron.run grid`（引数不足）で Grid 設定ダイアログが開き全リクエストが `Cannot perform '...' while a command dialog is open` で応答不能になった。存在しない部品名を指定した `Electron.run ADD <未解決名> ...` でも部品ブラウザがモーダルで開き同様にハングする。
- **`terminateActiveCommand` は「ダイアログが開いている」という理由で拒否され、MCP 側からの復旧手段が無い**。実測での唯一の復旧手段は AppleScript（System Events）でダイアログの Cancel/Done ボタンを直接クリックすることだった。GUI 操作なしにセッションを復旧できない = エージェント単独では回復不能な障害であり、運用上の重大リスクとして扱うこと。
- **ガードレール**: Electronics の書き込み系テキストコマンド（`ADD`, `NET`, `WIRE` など EAGLE の設計変更コマンド）を叩かない。生 EAGLE コマンドを実行する場合は、座標・単位などの引数が完備していると確認できるもの（`sch_placerectangle`/`sch_placeframe` のような座標指定の図形プリミティブコマンド）に限定する。
- **`Electron.run` 系の書き込みコマンドは silent no-op になることが多く、`OK` 相当の応答は成功を意味しない**。実行後は必ず `fusion_mcp_electronics_read` で対象エンティティ（`electronics.Part`/`electronics.Net`/`electronics.Wire` 等）の件数を照会し、意図した増分が実際に起きたかで成否判定すること。実測では `ADD`/`NET`/`WIRE` のいずれも応答は返るがエンティティ数が0のまま変化しない事例があった。
- **部品配置とネット配線は API 経由では非対応**。`adsk.electron` はオブジェクトレベルで完全 read-only（setter/add メソッド無し）であり、EAGLE ブリッジも図形プリミティブに限られる。回路設計の自動化を Fusion Electronics に期待しない。設計監査・ネット照会・BOM抽出などの読み取り用途に限定する（`tools.md` §4 参照）。

## トークン予算の目安（コミュニティ報告、参考値）

- 単一パーツ生成で 5 時間制限の約 13%、修正1回あたり約 7% というセッション消費の実測報告がある。一発完成を狙わず、修正前提で段階分解する（`SKILL.md` §3②と同じ結論）。
