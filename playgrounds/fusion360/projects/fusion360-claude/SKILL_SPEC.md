## 4. モデル・エージェント分担ポリシー（設計判断）

**結論: Fusion MCP を握るモデリングループは主ループ（セッションモデル = 現状 Fable 5）が直接実行する。「Fable 5 が設計・Sonnet 5 が実装」という固定分業はこのドメインでは採用しない。**

理由（批判的検討の要約）:
1. ボトルネックは実装の手数ではなく**空間推論と API の罠回避**（能力律速）。実装側を弱いモデルにするとリトライ回数が増える
2. リトライのコストが非対称に高い。失敗した CAD 操作は**共有可変状態（ドキュメント）を汚し**、undo 粒度・モード依存挙動・クラッシュのリスクを伴う。テキストコードの書き直しとは違う
3. 検証コストが高く委譲の利得を食う。オーケストレータがワーカーの Fusion 状態を検証するには結局スクリーンショット/エクスポートを自分で読む必要があり、「検証 ≒ 自分でやる」に近づく
4. 対象は単一ドキュメント + メインスレッド制約で**本質的に直列**。並列化の利得が構造的に出ない
5. 総コストはトークン単価でなく**往復回数**で決まる。安いモデルの 2–3 倍のリトライは高いモデルの一発より高くつき、コンテキストも汚す

**Sonnet 5 への委譲が正しい場面**（ステートレスで検証が安価な仕事）:
- 読み取り専用の調査・ドキュメント/API リファレンス探索（並列可）
- code-CAD ルート（build123d / forgecad）のコード生成。ヘッドレスで決定的に検証できるため弱いモデル + 強い検証器が成立する
- 主ループで一度実証済みのパラメタライズドスクリプトによる**量産バリアント**（20 サイズ展開等）
- レポート・ドキュメント整備

コンテキスト肥大の緩和は委譲ではなく手順で行う: スクリーンショットより数値クエリを優先、節目でエクスポートしてチェックポイント化。
