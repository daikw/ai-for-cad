## ハマった箇所と復旧

1. **コピー元 .eprj の残骸**: 空プロジェクトを作る API がないため既存 .eprj をコピーしたが、
   サンプルボードが同梱。`deleteBoard()` で消すと Schematic/PCB が「孤児ドキュメント」として
   残留 → `dmt_Schematic.deleteSchematic()` + `dmt_Pcb.deletePcb()` の個別削除で解消（+2 呼び出し）
2. **designator が自動採番されない**: create 直後は全部 "C?" 等。`setState_Designator` を
   toAsync/done パターンで 40 個明示設定して解決
3. **ピンオブジェクトの getter がプロトタイプに無い**: `getState_X` 等が無く、
   own property（`pin.x`, `pin.pinNumber`, `pin.rotation`）として直接生えていた。実体 introspection で発見
4. **save() しても .eprj の documents テーブルは旧内容のまま**: 編集内容は同ファイル内の
   history_data（op ログ、保存時刻と mtime 一致）に永続化され、documents.dataStr への
   materialize はプロジェクトクローズ等まで遅延される模様。永続化確認は「doc close→reopen→再カウント」で実施
5. （非再現）メモリで警告されていた **22pF C1653 の create ハングは今回発生せず**。
   BOM の device uuid が対策済みだった。ただし警告に従い 22pF を最初に単独 execute で
   配置してリスクを局所化した（ハング時の被害 = バッチ全体ではなく 1 呼び出し）

## 前回蓄積（スキル・メモリ）の効き — ツーリング成熟度の比較材料

**初見なら確実に詰まっていた箇所**（メモリ/スキルで即回避できたもの）:

- `createProject` がオフラインで **silent fail** する仕様。「cp → sqlite で projects.name UPDATE →
  openProject(sha256(ファイルパス))」という非自明な回避策はメモリ由来。初見なら数時間級の行き止まり
- **スタブワイヤ + ネット名で接続が成立する**という配線モデル。初見ならピン間の
  グラフィカル配線経路を計算しようとして大幅に遠回りしたはず
- 回路図の座標単位 = **0.01 inch**（PCB は 1mil）。スキルの CRITICAL 警告どおり、初見の最頻ミス
- ERC の正しい呼び出し形 `sch_Drc.check(false, false, true)`（verbose 配列が返る）
- 22pF ハング警告 → 配置順序の防御的設計
- ブリッジのポートスキャン手順（49620–49629 + handshake 検証）

**今回新たに得た知見**（メモリ更新候補）: 上記ハマり 1–4。特に「.eprj は op ログ永続化で
documents materialize は遅延」は保存検証の設計に直結する。

## エージェント操作性の評価（API 成熟度）

| 領域 | 評価 | 所感 |
|---|---|---|
| ライブラリ | ○（今回ほぼ未使用） | device uuid 直指定の create は堅牢。uuid さえ確定していれば検索・選定コストゼロ |
| 配置 | ◎ | `sch_PrimitiveComponent.create({libraryUuid, uuid}, x, y)` は 40/40 成功。難点: designator 非自動採番、ピン情報 API の型ドキュメント不足（getter 不在） |
| 配線 | ◎ | `sch_PrimitiveWire.create(line, net)` + ネットラベル意味論が宣言的配線に最適。エージェント適性は GUI 配線より圧倒的に高い。162/162 一発成功 |
| 検証 | ◎ | `.enet` が JSON（components + pinInfoMap）で機械可読、ERC も構造化配列で返る。幾何ダンプ（ピン座標・ワイヤ座標）も完備で独立検証を組める。今回最も感心した領域 |
| プロジェクト管理 | △ | 最弱。オフラインで createProject 不能（sqlite 工作必須）、deleteBoard の孤児化、closeProject 不在、save の materialize 遅延が不透明 |

## 比較上の最重要発見 3 点

1. **2 回目のランは初回の桁違いに速い**: 初回（7/2–3）は回路図 + PCB 全体で丸 1 日超、
   今回は回路図を約 20 分・28 execute・エラー 0 で完遂。差分はほぼ全てスキル文書 + メモリの
   蓄積（プロジェクト作成ハック、配線モデル、単位系、ERC 呼び出し形）。エージェント CAD の
   生産性はツール自体より「攻略知識の永続化」に支配される
2. **ネットラベル意味論がエージェント配線の勝ち筋**: 見た目のワイヤルーティングを完全に
   放棄し「ピン座標に 0.1 inch のスタブ + ネット名」に還元すると、配線問題が宣言的な
   pin→net 表（コードレビュー可能・自動検証可能）になる。回路図の美観と引き換えに
   正確性と速度が桁で上がる
3. **検証パイプラインが API で完結する**: ピン/ワイヤ幾何の全量ダンプ → ローカル union-find、
   加えてネイティブ .enet 出力（JSON）との二重照合、ERC の構造化結果取得まで全て
   ヘッドレスで可能。「エージェントが自分の作業を自分で監査できる」度合いは
   このベンチで比較する価値のある軸
